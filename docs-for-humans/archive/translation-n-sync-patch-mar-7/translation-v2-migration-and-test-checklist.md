## Миграция IndexedDB (одноразовая, принудительная)

Причина: старый кэш мог содержать “кашу” (fallback/оригинал, ошибочно принятый за target язык). Для Translation v2 это недопустимо.

Решение: bump версии IndexedDB и очистка stores, которые связаны с контентом/переводами.

Где:
- `src/lib/contentCache.ts` — `openDb()` / `onupgradeneeded`

Что делаем:
1) Увеличить `DB_VERSION`.
2) В `onupgradeneeded` выполнить `clear()` для:
   - `STORE_CHAPTER_SKELETON`
   - `STORE_BLOCK_TEXT`
   - (и legacy store, если он ещё существует)
3) Не трогать unrelated stores (если есть).

Важно:
- это именно **одноразовая миграция релиза**, а не постоянная runtime-логика;
- цель миграции: удалить старые mixed-language / poisoned cache entries, где fallback мог быть ошибочно принят за target text.

Ожидаемое поведение у пользователей:
- при первом открытии после деплоя создаётся новая версия IDB и кэш контента очищается автоматически.

## Чеклист тестов (обязательно)

### A. Базовый рендер языка
1) Открыть книгу, выбрать RU/FR/ES.
2) Пока перевода нет → блоки обязаны быть под blur, но **не пустые**.
3) По мере появления `Text[blockId, activeLang]` blur снимается частично/постепенно.
4) Наличие любого fallback текста не должно снимать blur.

### B. Уход/возврат (главный регресс)
1) Включить target язык, дождаться старт переводов.
2) Уйти в Library.
3) Вернуться в Reader.
Ожидание:
- UI не показывает “не target” без blur.
- reconcile добирает missing переводы для видимого окна.
- возврат не зависит от того, дослушал ли клиент старый стрим до конца.
- если старый `pending` протух и превратился в `missing`, recovery path не должен молча терять блок: должен происходить retry translate или обычный re-enqueue до terminal state.

### C. Jump внутри ридера
1) Включить target язык.
2) Прыжок по TOC в другую главу.
3) Сразу прыжок обратно.
Ожидание:
- старые in-flight запросы могут cancel/abort, но итоговый рендер корректен;
- reconcile/окна не оставляют “не target” без blur.
- IDB-first логика не запрашивает заново блоки, которые уже есть локально для target-языка.

### D. Быстрое листание
1) Включить target язык.
2) Быстро листать страницы (Next page) 10–20 раз.
Ожидание:
- HIGH приоритет всегда переводит видимые блоки;
- LOW/EXTRALOW не мешают (без лавины запросов/подвисаний).
- отсутствие `Text[blockId, activeLang]` всегда означает blur, независимо от того, что уже лежит в fallback.

### E. Межглавный prefetch
1) На конце главы листать дальше.
Ожидание:
- prefetch окно “добирает” блоки из следующей главы, пока не наберёт целевое число блоков.
- перед запросом на сервер сначала отбрасываются blockIds, уже существующие в IDB для `activeLang`.

### F. Смена языка на уже частично закешированный
1) Открыть книгу и частично дождаться RU.
2) Переключиться на EN, потом обратно на RU.
Ожидание:
- блоки, у которых RU уже есть локально, сразу отображаются без blur;
- остальные показывают fallback под blur;
- reconcile добирает только missing RU-блоки.

### G. Проверка старых endpoint-ов
1) Проверить, что Reader не зависит от старого `translate-status` как от отдельной истины ready-state.
2) Проверить, что `/content?lang=XX` не используется как частый reconcile-механизм.
Ожидание:
- bulk `POST /blocks/text` используется для догонки по окнам;
- `/translate` остается стримовым push-механизмом;
- `/content` остается snapshot/skeleton/fallback-загрузкой.

## Что мониторить в Network (практика)

- `GET /content?lang=XX` — только при загрузке снапшота/скелета.
- `POST /translate` — стриминг результатов (может abort).
- `POST /blocks/text` (reconcile) — bulk fetch missing переводов по ids.

Главный критерий успеха:
- при выбранном RU/FR/ES не должно быть “EN без blur”.
