# План Оптимизации Оркестрации Переводов И Догрузки

## Контекст
Сейчас перевод приходит по `blockId` и коммитится на клиент по одному блоку. Это даёт быстрый частичный отклик, но создаёт рваные состояния на странице:
- часть строк уже переведена,
- соседние блоки ещё pending,
- частые микро-репагинации,
- визуально нестабильный первый читаемый экран.

Нужна политика, которая балансирует:
1. быстрое появление читаемой страницы,
2. стабильность вёрстки,
3. эффективность backend-вызовов.

## Цели
1. Снизить время до `First Readable Page`.
2. Убрать рваный коммит «один блок готов, соседний нет» в пределах видимого окна.
3. Снизить количество лишних репагинаций.
4. Повысить эффективность запросов (меньше overhead на мелких блоках).

## Нецели
1. Полный отказ от stream-ответов.
2. Полный отказ от block-level модели хранения.
3. Переписывание paginator-движка в рамках этого этапа.

## Проблемы Текущего Подхода
1. `MAX_BATCH_SIZE` ограничивает число блоков, но не контролирует реальный токен-объём.
2. Очень короткие блоки обрабатываются так же, как большие, хотя их рациональнее агрегировать.
3. UI коммитит «по мере прихода каждого блока», что создаёт визуальный шум.
4. Репагинация может запускаться слишком часто при микропорциях текста.

## Целевая Модель

### 1) Единица Планирования: Adaptive Translation Chunk
Вместо жёсткого «N blockIds за раз» вводим `chunk`:
- chunk = последовательность соседних blockIds,
- собирается до лимита по токенам/символам,
- учитывает типы блоков (heading/list/paragraph).

Правила агрегации:
1. Склеивать короткие соседние текстовые блоки.
2. Не склеивать через структурные разрывы (image/hr/section boundary).
3. Держать отдельные chunk-правила для list (чтобы не терять структуру).
4. Верхний лимит размера: `max_tokens_per_chunk`.

### 2) Приоритезация Очередей
Три приоритета:
1. `P0` — текущая видимая страница (single) или текущий spread.
2. `P1` — следующая страница (и опционально предыдущая для back navigation).
3. `P2` — остальной хвост главы.

Политика отмен:
1. При навигации на новую страницу активный `P0` старой страницы понижается/прерывается.
2. В работе остаётся только актуальный `P0` текущего viewport.

### 3) Политика Коммита На Клиент
Коммитить не «каждый блок сразу», а staged-окнами:
1. `Stage A (First Readable)`:
- коммит минимального набора, делающего страницу читаемой,
- критерий: на странице есть завершённый связный текстовый сегмент без разрывов между соседними short-block.
2. `Stage B (Viewport Complete)`:
- коммит всего `P0`-окна батчем.
3. `Stage C (Prefetch)`:
- коммит `P1/P2` по мере готовности.

### 4) Политика Репагинации
1. Не репагинировать на каждый микрорезультат.
2. Ввести debounce + coalesce окно (например, 80-150мс) для нескольких переводов.
3. Для `P0` допускается ускоренный commit, но всё равно с микробатчем.

## Backend Контракт (Предложение)

### Новый endpoint (или режим)
`POST /api/chapters/:id/translate-window`

Request:
- `lang`
- `anchorBlockId`
- `direction`
- `targetWindowBlockIds` (или page window descriptor)
- `maxTokensPerChunk`
- `maxChunksInFlight`

Response stream events:
1. `chunk_started`
- `chunkId`
- `blockIds`
- `priority`
2. `chunk_result`
- `chunkId`
- `translations: [{ blockId, payload }]`
- `status`
3. `chunk_failed`
- `chunkId`
- `retryable`
- `reason`
4. `translate_done`
- summary metrics

Обязательное:
1. Идемпотентность по `requestId/chunkId`.
2. Стабильный mapping `chunk -> blockIds`.
3. Частичный retry без пересборки всей задачи.

## Frontend Оркестратор (Изменения)
1. Добавить `TranslationScheduler` поверх текущей очереди.
2. Планировщик формирует adaptive chunks из `P0/P1/P2`.
3. Отдельно хранить:
- `chunk state`,
- `block state`,
- `ui-commit state`.
4. Обновлять `displayBlocks` батчами по chunk (или mini-batch), а не по каждому блоку немедленно.
5. Репагинацию запускать через coalesce-буфер.

## Политика Для Коротких Блоков
1. Если подряд идут короткие paragraph/quote/heading, объединять в один chunk.
2. Если суммарный объём ниже порога, не отправлять их как отдельные 1-line запросы.
3. Для list:
- можно агрегировать весь список в один chunk,
- возвращать по item mapping, чтобы не ломать структуру.

## Кэширование
1. Кэш на уровне chunk (ключ: `chapterId + lang + modelVersion + chunkHash`).
2. Кэш на уровне block оставляем для совместимости.
3. При cache hit на chunk — немедленный batched commit в UI.

## Метрики (минимум)
1. `time_to_first_readable_page_ms`
2. `time_to_viewport_complete_ms`
3. `repagination_runs_per_min`
4. `avg_chunk_tokens`
5. `avg_blocks_per_chunk`
6. `abort_rate_by_priority`
7. `partial_viewport_ratio` (доля времени, когда viewport не полностью готов)

## Пошаговый Rollout

### Этап 1: Быстрые улучшения без смены API
1. На клиенте вводим coalesced commit + coalesced repagination.
2. Агрегируем короткие block updates перед `setDisplayBlocks`.
3. Меряем baseline/после.

### Этап 2: Backend chunk contract
1. Добавляем `translate-window` с chunk событиями.
2. Подключаем новый scheduler под feature flag.
3. A/B на части трафика.

### Этап 3: Оптимизация и стандартизация
1. Тюнинг `maxTokensPerChunk` по метрикам.
2. Тюнинг P0/P1 window size.
3. Финализация SLA для first readable viewport.

## Риски И Митигации
1. Риск: chunk слишком большой -> длинный TTFP.
- Митигация: hard cap токенов + ранний `Stage A`.
2. Риск: сложный mapping chunk->blocks.
- Митигация: строгий schema contract + валидация на клиенте.
3. Риск: регрессии в anchor restore.
- Митигация: отдельные e2e сценарии для language switch и page jump.

## Критерии Готовности
1. На проблемных книгах исчезают рваные «1 строка переведена, соседняя нет» для `P0` окна.
2. Снижается количество репагинаций на активном чтении.
3. `time_to_first_readable_page` не ухудшается, а в идеале уменьшается.
4. Нет регрессий в anchor restore и page navigation.
