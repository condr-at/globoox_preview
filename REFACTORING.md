# Анализ и предложения по рефакторингу

## Часть 1. Архитектура и критичные проблемы

### Боттлнеки

| Область | Суть |
|---------|------|
| **Перевод и рендер** | Частичное обновление UI по мере стрима: «лоскутный» список/страница, blur + «Translating…», высокий CLS. Долгота первого экрана (10–20 с) зависит не только от сети/LLM: критична политика **когда** коммитить готовые фрагменты в DOM. |
| **Кеш перевода** | Привязка к `blockId`/`chapterId` с бэкенда: смена парсинга или миграции → промахи кеша и лишние повторные переводы. |
| **Кросс-девайс** | Синк версий библиотеки/прогресса есть; **переиспользование готовых переводов на втором устройстве** отдельным слоем по сути не закрыто. |
| **Комплаенс / продукт** | Жёсткое ограничение скачивания/правки в браузере — веб слаб; сильнее нативная оболочка. E2EE на сервере не убирает plaintext у внешнего API перевода — отдельное решение. |
| **Кросс-платформа** | «Как нативка» без смены стека: оболочка (Capacitor / Tauri / RN) + общее ядро. |
| **«Шифруем на сервере»** | Зафиксировать в backend/infra (at-rest, ключи, бэкапы) для audit story — вне скоупа фронта, но для клиента важно. |

### Направления работ (приоритет)

**A — высокий эффект, низкий риск (логика перевода / отображения)**

- Атомарный commit перевода для **видимой группы** (экран TOC / страница ридера), не подмешивание по одному блоку в длинный список.
- **Coalesce** обновлений из стрима (например, 100–200 ms) перед ре-рендером.
- Стабилизация лэйаута при загрузке: placeholder / одношаговая замена слоя, не покусочная подстановка текста.
- **Prefetch** следующего экрана в рамках бюджета стоимости и rate limit.

**B — эффективность и второе устройство**

- Контент-адресный ключ кеша (нормализация текста + книга + lang + версия модели), устойчивый к смене block id.
- Явный **синк сегментов перевода** на сервер (формат и шифрование — по продукту/юристам).
- Eviction / лимиты IndexedDB (LRU по книге или глобально).

**C — по ROI продукту**

- Нативная оболочка + вынос ядра в пакет.
- Клиентский ключ + ciphertext на сервере при необходимости позиции «сервер не читает контент».

### Стратегия и метрики

Полный rewrite стека **не оправдан**. Путь: **A → B**; **C** отдельно по ROI.  
Метрики приёмки: время до первой читаемой переведённой «страницы», CLS, доля cache hit при повторном чтении, число LLM-батчей на книгу / язык.

## Часть 2. UI/UX и улучшения

### Дизайн-система и светлая тема

- В корне уже есть `next-themes` (`defaultTheme="system"`), токены в `globals.css`, но в ридере/шитах много **жёстких цветов** (`#f3f3f1`, `#1c1c1e` и т.д.) вместо семантических переменных — светлая тема **фактически не доведена** или экраны выглядят «всегда тёмными».
- **Предложение:** аудит ридера и модалок → перевод на токены; переключатель **Светлая / Тёмная / Как в системе** в настройках с единым источником правды (`next-themes` + store, без расхождений); отдельная проверка контраста и `:focus-visible` в светлой теме.

### Оглавление (TOC)

- «Лишние главы» обычно из модели данных: смешение **spine** и **nav**, один XHTML = «глава», дубли nav/NCX, отсутствие фильтра пустых/служебных узлов.
- **Предложение:** строить TOC из **EPUB3 nav** (fallback NCX), spine — с правилами отсечения; дедуп по `href`; опционально дерево глубиной и режим «технические разделы» для отладки.

### Пагинация и жесты

- У EPUB **нет** фиксированных «страниц издателя» как у PDF; страница — результат вёрстки (шрифт, ширина). `page-list` встречается редко.
- Паттерн **вертикальный скролл внутри экрана + горизонтальный свайп (моб.) / стрелки (десктоп)** — уместен для читалки и упорядочивает **момент запроса** перевода по видимому экрану.
- **Стабильность чанков для кеша LLM:** привязка к **логическим единицам текста** (блок/сегмент + контент-ключ) надёжнее, чем к «странице экрана» — при смене шрифта/ориентации границы экрана меняются. Экран хорош как триггер prefetch и размера **батча запроса**, не как единственный id кеша.

### Прочее (вне ядра функционала)

- **a11y:** контраст, фокус, размер тап-зон в шитах.
- **Согласованность:** смесь iOS-like шитов и shadcn — зафиксировать правила (когда какой паттерн).
- **Длинный TOC:** виртуализация списка при очень больших книгах.
- **PWA / оффлайн:** если в позиционировании важен оффлайн — явный статус и политика хранения EPUB на клиенте (сейчас упор на онлайн + IndexedDB-кеши).
